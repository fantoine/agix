## `remove` / `update` ŌĆö mangled lock source aborts uninstall (Task 4 review)

**Scenario:** Pre-existing lock entry whose `source` string doesn't parse via `SourceSpec::parse`.
**Command:** `agix remove <pkg>` or `agix update`
**Expected:** warn and fall back to file-based uninstall so user can still clean up.
**Actual:** `Installer::uninstall` in `src/core/installer.rs:184` does `SourceSpec::parse(&pkg.source)?` ŌĆö any parse error aborts before any files are removed or the lock entry is cleaned.
**Severity:** minor (would only bite on hand-edited or older-format lockfiles).
**Fix commit:** Deferred to Task 15 (review `remove`).

## `add marketplace` ŌĆö total failure returns Ok (Task 4 review)

**Scenario:** `agix add marketplace <bad-ref>` where every target CLI fails.
**Command:** `agix add marketplace <bad/ref>@<plugin>`
**Expected:** non-zero exit (CI pipelines should detect failure).
**Actual (before fix):** `installer.rs:35-97` turned each per-driver failure into `output::warn` and continued; `install_manifest` returned `Ok(())` even if no driver succeeded. Contrast with non-marketplace branch which propagates via `?`.
**Resolution:** `install_manifest` now tracks a `success_count` in the marketplace branch. If `target_clis` was non-empty and no driver succeeded, the function returns `AgixError::Other("marketplace plugin '{plugin}' from '{marketplace}' failed to install for all target CLIs")`. The historical "no detected CLIs ŌåÆ target_clis is empty ŌåÆ Ok" path is preserved (warnings only). Regression test: `regression_marketplace_total_failure_returns_nonzero` in `tests/add_test.rs` uses a `claude` shim that exits non-zero for every invocation.
**Severity:** minor (affects CI ergonomics only; warnings still visible).
**Fix commit:** `90bb91a`

## `add --scope global` ŌĆö auto-init always prompts on fresh machine (Task 5 review)

**Scenario:** User on a fresh machine runs `agix add --scope global github:org/pkg`; no global Agentfile exists yet so `agentfile_paths` auto-creates one.
**Command:** `agix add --scope global ...` (or any command triggering global auto-init)
**Expected:** respect the calling command's non-interactive state ŌĆö if the user could pass `--no-interactive` to `add`, the auto-init should honor it too.
**Actual (before fix):** `src/commands/mod.rs:33-41` hard-coded `pick_clis(&[], false)` when auto-initializing. The only escape hatch was the `AGIX_NO_INTERACTIVE` env var (checked inside `pick_clis`); CI pipelines without that env var would block on a TTY prompt.
**Resolution:** `agentfile_paths` now takes an explicit `non_interactive: bool` parameter and forwards it to `pick_clis`. Call-sites in `add`/`install`/`remove`/`update`/`list`/`outdated` pass `false` today (none of them accept a `--no-interactive` flag yet), which preserves their current behaviour. The functional escape hatch remains `AGIX_NO_INTERACTIVE=1`, checked inside `pick_clis`. The value of the change is plumbing: when any of those commands grow an explicit non-interactive flag (Task 18 cross-cutting sweep), the seam exists and `agentfile_paths` will honour it without another refactor. Regression test: `regression_global_auto_init_honors_non_interactive_env_var` in `tests/add_test.rs` runs `add --scope global` against a fresh `HOME` with `AGIX_NO_INTERACTIVE=1` and asserts completion without TTY. Documenting / auto-detecting `AGIX_NO_INTERACTIVE` remains deferred to Task 18.
**Severity:** minor (affects non-interactive `add`/`remove`/`update`/`install` UX on first global call).
**Fix commit:** `90bb91a`

## `check` ŌĆö unknown CLI names silently accepted (Task 9 review)

**Scenario:** `[agix].cli` lists a CLI name with no registered driver (e.g. `cli = ["not-a-real-cli"]`).
**Command:** `agix check`
**Expected:** surface the fact visibly so the user can correct a typo, but do not hard-fail ŌĆö `check` is a structural validator and the user may be preparing a manifest for a CLI they plan to support later.
**Actual (before fix):** command reported `Agentfile valid ŌĆö project for not-a-real-cli` with no signal at all; typos were invisible until `install` / `doctor` later.
**Decision:** warn (not error). Enforced via `drivers::driver_for(name)` with the list of known drivers mentioned in the warning text for discoverability.
**Severity:** minor (UX; no functional bug).
**Fix commit:** see `review(check):` commit on this branch.

## `add` ŌĆö unknown CLI in `--cli` silently accepted (Task 10 review)

**Scenario:** `agix add local ./pkg --cli not-a-real-cli` on an Agentfile where `[agix].cli` does not mention `not-a-real-cli`.
**Command:** `agix add <type> <value> --cli <typo>`
**Expected:** exit non-zero; error points at the typo and lists known drivers so the user can correct it.
**Actual (before fix):** the bogus CLI name was written straight into `[not-a-real-cli.dependencies]`. No driver owns that section, so the install silently did nothing and the manifest was corrupted.
**Decision:** error (stricter than `check`, which warns). `add` is a write operation and a typo would otherwise persist in the user's TOML.
**Severity:** minor (no data loss, but silent no-op is worse than a clear failure).
**Fix commit:** `90bb91a`

## `add` ŌĆö duplicate add silently overwrote the existing entry (Task 10 review)

**Scenario:** User runs `agix add local ./pkg` twice (or `agix add ... --cli claude` twice). The second call silently replaced the first dependency without any signal.

**Fix commit:** `90bb91a`

## Cross-cutting ŌĆö integration tests leak into real `$HOME` (Task 7 review)

**Scenario:** `agix install` / `agix add` / `agix export` integration tests run without overriding `HOME`, so the `claude-code` driver's install step writes into the user's real `~/.claude/agix/` and the `codex` driver into `~/.codex/agix/`.

**Severity:** minor (works in practice, but surprising and occasionally destructive).
**Fix commit:** Deferred to Task 18 (cross-cutting sweep, Phase B).
