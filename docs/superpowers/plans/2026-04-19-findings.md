## `remove` / `update` — mangled lock source aborts uninstall (Task 4 review)

**Scenario:** Pre-existing lock entry whose `source` string doesn't parse via `SourceSpec::parse`.
**Command:** `agix remove <pkg>` or `agix update`
**Expected:** warn and fall back to file-based uninstall so user can still clean up.
**Actual:** `Installer::uninstall` in `src/core/installer.rs:184` does `SourceSpec::parse(&pkg.source)?` — any parse error aborts before any files are removed or the lock entry is cleaned.
**Severity:** minor (would only bite on hand-edited or older-format lockfiles).
**Fix commit:** Deferred to Task 15 (review `remove`).

## `add marketplace` — total failure returns Ok (Task 4 review)

**Scenario:** `agix add marketplace <bad-ref>` where every target CLI fails.
**Command:** `agix add marketplace <bad/ref>@<plugin>`
**Expected:** non-zero exit (CI pipelines should detect failure).
**Actual:** `installer.rs:35-97` turns each per-driver failure into `output::warn` and continues; `install_manifest` returns `Ok(())` even if no driver succeeded. Contrast with non-marketplace branch which propagates via `?`.
**Severity:** minor (affects CI ergonomics only; warnings still visible).
**Fix commit:** Deferred to Task 10 (review `add`).

## `install` — `claude plugin install` on already-installed plugin warns non-zero (Task 4 review)

**Scenario:** Re-run `agix install` on a manifest with a marketplace plugin already installed via `claude`.
**Command:** `agix install`
**Expected:** silent success (idempotent).
**Actual:** `claude plugin install <x>@<y>` exits non-zero when already installed; `ClaudeCodeDriver::install_marketplace_plugin` bubbles this as `AgixError::Other`, which the installer surfaces as `warn`.
**Severity:** cosmetic (correct state is reached; noise only).
**Fix commit:** Deferred to Task 12 (review `install`).

## `doctor` — marketplace packages silently pass (Task 4 review)

**Scenario:** Marketplace-installed plugin files deleted manually from `~/.claude/plugins/`.
**Command:** `agix doctor`
**Expected:** warn that the plugin is missing or at least note `marketplace (managed by <cli>)`.
**Actual:** `src/commands/doctor.rs:24-30` iterates `pkg.files`, which is empty for marketplace packages — loop is a no-op, doctor reports "All installed files present".
**Severity:** minor (misleading but not incorrect — we don't own those files).
**Fix commit:** Deferred to Task 13 (review `doctor`, Phase B).

## `add --scope global` — auto-init always prompts on fresh machine (Task 5 review)

**Scenario:** User on a fresh machine runs `agix add --scope global github:org/pkg`; no global Agentfile exists yet so `agentfile_paths` auto-creates one.
**Command:** `agix add --scope global ...` (or any command triggering global auto-init)
**Expected:** respect the calling command's non-interactive state — if the user could pass `--no-interactive` to `add`, the auto-init should honor it too.
**Actual:** `src/commands/mod.rs:33-41` hard-codes `pick_clis(&[], false)` when auto-initializing. The only escape hatch is the `AGIX_NO_INTERACTIVE` env var (checked inside `pick_clis`). CI pipelines without that env var will block on a TTY prompt.
**Severity:** minor (affects non-interactive `add`/`remove`/`update`/`install` UX on first global call).
**Fix commit:** Deferred to Task 10 (review `add`, Phase B).

## Cross-cutting — `AGIX_NO_INTERACTIVE` is undocumented global state (Task 5 review)

**Scenario:** Users scripting agix in CI must discover `AGIX_NO_INTERACTIVE` from source or tests.
**Expected:** either document the env var in `--help` / README, or auto-detect non-interactive context (e.g. `!console::Term::stderr().is_term()` or `CI` env var).
**Actual:** only referenced inline in `src/ui/prompt.rs`; no user-facing mention; `--no-interactive` only exists on `init`.
**Severity:** minor (UX/docs; no functional bug).
**Fix commit:** Deferred to Task 18 (cross-cutting sweep, Phase B).

## Cross-cutting — no unified test helper for non-interactive mode (Task 5 review)

**Scenario:** Every integration test touching init/add/install must remember to either pass `--no-interactive` or set `AGIX_NO_INTERACTIVE=1`; easy to forget in new tests and deadlocks CI.
**Expected:** a shared `tests/helpers/mod.rs` helper (e.g. `cmd_non_interactive()` returning a pre-configured `assert_cmd::Command`) so new tests default to safe behavior.
**Actual:** `.env("AGIX_NO_INTERACTIVE", "1")` and `--no-interactive` flags are sprinkled across `tests/init_test.rs` and `tests/add_test.rs` without a common helper.
**Severity:** minor (maintenance time bomb; first missed usage blocks CI).
**Fix commit:** Deferred to Task 18 (cross-cutting sweep, Phase B).

**Severity:** minor (valid TOML still, but subsequent commands iterate the driver twice).

## `init` — non-interactive without TTY surfaces opaque error (Task 8 review)

**Scenario:** CI or piped-stdin caller runs bare `agix init` (no `--no-interactive` flag, no `AGIX_NO_INTERACTIVE` env var).
**Command:** `echo "" | agix init`
**Expected:** clear error pointing at `--no-interactive` / `AGIX_NO_INTERACTIVE`.
**Actual:** exits 1 with `Error: prompt failed: IO error: not a terminal`. No deadlock (good), but the message doesn't hint at the escape hatches. Related to the already-deferred "undocumented `AGIX_NO_INTERACTIVE`" finding.
**Severity:** cosmetic (UX/docs; functional behavior is correct).
**Fix commit:** Deferred to Task 18 (cross-cutting sweep) — will be resolved when `AGIX_NO_INTERACTIVE` is documented / auto-detected.

## `check` — unparseable dependency sources silently pass (Task 9 review)

**Scenario:** Agentfile declares a dependency whose source has an unknown / missing scheme (e.g. `my-dep = "nope:whatever"`), either under `[dependencies]` or under `[<cli>.dependencies]`.
**Command:** `agix check`
**Expected:** exit non-zero; error message names the offending dep so the user can locate it.
**Actual (before fix):** `src/commands/check.rs` never called `sources::parse_source`; the command only validated `[agix]` fields and reported `Agentfile valid`. A follow-up `agix install` would be the first place the user learned the source was bogus.
**Severity:** minor (hidden validation gap; caught at install time anyway).
**Fix commit:** see `review(check):` commit on this branch.

## `check` — unknown CLI names silently accepted (Task 9 review)

**Scenario:** `[agix].cli` lists a CLI name with no registered driver (e.g. `cli = ["not-a-real-cli"]`).
**Command:** `agix check`
**Expected:** surface the fact visibly so the user can correct a typo, but do not hard-fail — `check` is a structural validator and the user may be preparing a manifest for a CLI they plan to support later.
**Actual (before fix):** command reported `Agentfile valid — project for not-a-real-cli` with no signal at all; typos were invisible until `install` / `doctor` later.
**Decision:** warn (not error). Enforced via `drivers::driver_for(name)` with the list of known drivers mentioned in the warning text for discoverability.
**Severity:** minor (UX; no functional bug).
**Fix commit:** see `review(check):` commit on this branch.

## Cross-cutting — integration tests leak into real `$HOME` (Task 7 review)

**Scenario:** `agix install` / `agix add` / `agix export` integration tests run without overriding `HOME`, so the `claude-code` driver's install step writes into the user's real `~/.claude/agix/` and the `codex` driver into `~/.codex/agix/`.
**Expected:** tests should set `HOME` to a tempdir (or redirect via a dedicated test harness) so suites are hermetic and safe to run on dev machines.
**Actual:** existing integration tests (notably `tests/add_test.rs`, `tests/install_test.rs`, `tests/export_roundtrip_test.rs`) rely on the host `$HOME`. A developer running `cargo test` pollutes their own Claude Code / Codex install directories.
**Severity:** minor (works in practice, but surprising and occasionally destructive).
**Fix commit:** Deferred to Task 18 (cross-cutting sweep, Phase B).
