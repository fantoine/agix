## `remove` / `update` — mangled lock source aborts uninstall (Task 4 review)

**Scenario:** Pre-existing lock entry whose `source` string doesn't parse via `parse_source`.
**Command:** `agix remove <pkg>` or `agix update`
**Expected:** warn and fall back to file-based uninstall so user can still clean up.
**Actual (before fix):** `Installer::uninstall` in `src/core/installer.rs` called `parse_source(&pkg.source)?` — any parse error aborted before any files were removed or the lock entry was cleaned.
**Resolution:** `Installer::uninstall` now catches `parse_source` errors, emits `output::warn("lock source unparseable for '{name}': {e}; falling back to file-based uninstall")`, and treats the package as non-marketplace (uses `pkg.files` to delete and removes from lock). When the degraded path has no tracked files, a second warning reminds the user to clean up marketplace-managed files manually via the relevant CLI. Regression test: `regression_mangled_lock_source_falls_back_and_warns` in `tests/remove_test.rs` crafts a lock entry with an unparseable source + tracked file and asserts the command succeeds, the file gets deleted, and the lock entry is dropped.
**Severity:** minor (would only bite on hand-edited or older-format lockfiles).
**Fix commit:** see `review(remove):` commit on this branch.

## `add marketplace` — total failure returns Ok (Task 4 review)

**Scenario:** `agix add marketplace <bad-ref>` where every target CLI fails.
**Command:** `agix add marketplace <bad/ref>@<plugin>`
**Expected:** non-zero exit (CI pipelines should detect failure).
**Actual (before fix):** `installer.rs:35-97` turned each per-driver failure into `output::warn` and continued; `install_manifest` returned `Ok(())` even if no driver succeeded. Contrast with non-marketplace branch which propagates via `?`.
**Resolution:** `install_manifest` now tracks a `success_count` in the marketplace branch. If `target_clis` was non-empty and no driver succeeded, the function returns `AgixError::Other("marketplace plugin '{plugin}' from '{marketplace}' failed to install for all target CLIs")`. The historical "no detected CLIs → target_clis is empty → Ok" path is preserved (warnings only). Regression test: `regression_marketplace_total_failure_returns_nonzero` in `tests/add_test.rs` uses a `claude` shim that exits non-zero for every invocation.
**Severity:** minor (affects CI ergonomics only; warnings still visible).
**Fix commit:** `90bb91a`

## `install` — `claude plugin install` on already-installed plugin warns non-zero (Task 4 review)

**Scenario:** Re-run `agix install` on a manifest with a marketplace plugin already installed via `claude`.
**Command:** `agix install`
**Expected:** silent success (idempotent).
**Actual (before fix):** `claude plugin install <x>@<y>` exits non-zero when already installed; `ClaudeDriver::install_marketplace_plugin` bubbled this as `AgixError::Other`, which the installer surfaced as `warn`.
**Resolution:** `ClaudeDriver::install_marketplace_plugin` now captures stdout/stderr via `Command::output()` (instead of `.status()`) and checks the combined output against a small, case-insensitive phrase list (`already installed`, `already exists`, `already added`) before raising an error. The same shield applies to `claude plugin marketplace add`. Phrases are intentionally narrow so genuine failures still surface. Regression test: `regression_marketplace_reinstall_already_installed_is_silent_success` in `tests/install_test.rs` uses a `claude` shim that exits 1 with "Plugin already installed" on stderr and asserts the command succeeds and does not emit an `install failed for` warning.
**Severity:** cosmetic (correct state is reached; noise only).
**Fix commit:** `e9eea86`

## `add --scope global` — auto-init always prompts on fresh machine (Task 5 review)

**Scenario:** User on a fresh machine runs `agix add --scope global github:org/pkg`; no global Agentfile exists yet so `agentfile_paths` auto-creates one.
**Command:** `agix add --scope global ...` (or any command triggering global auto-init)
**Expected:** respect the calling command's non-interactive state — if the user could pass `--no-interactive` to `add`, the auto-init should honor it too.
**Actual (before fix):** `src/commands/mod.rs:33-41` hard-coded `pick_clis(&[], false)` when auto-initializing. The only escape hatch was the `AGIX_NO_INTERACTIVE` env var (checked inside `pick_clis`); CI pipelines without that env var would block on a TTY prompt.
**Resolution:** `agentfile_paths` now takes an explicit `non_interactive: bool` parameter and forwards it to `pick_clis`. Call-sites in `add`/`install`/`remove`/`update`/`list`/`outdated` pass `false` today (none of them accept a `--no-interactive` flag yet), which preserves their current behaviour. The functional escape hatch remains `AGIX_NO_INTERACTIVE=1`, checked inside `pick_clis`. The value of the change is plumbing: when any of those commands grow an explicit non-interactive flag (Task 18 cross-cutting sweep), the seam exists and `agentfile_paths` will honour it without another refactor. Regression test: `regression_global_auto_init_honors_non_interactive_env_var` in `tests/add_test.rs` runs `add --scope global` against a fresh `HOME` with `AGIX_NO_INTERACTIVE=1` and asserts completion without TTY. Documenting / auto-detecting `AGIX_NO_INTERACTIVE` remains deferred to Task 18.
**Severity:** minor (affects non-interactive `add`/`remove`/`update`/`install` UX on first global call).
**Fix commit:** `90bb91a`

## `check` — unknown CLI names silently accepted (Task 9 review)

**Scenario:** `[agix].cli` lists a CLI name with no registered driver (e.g. `cli = ["not-a-real-cli"]`).
**Command:** `agix check`
**Expected:** surface the fact visibly so the user can correct a typo, but do not hard-fail — `check` is a structural validator and the user may be preparing a manifest for a CLI they plan to support later.
**Actual (before fix):** command reported `Agentfile valid — project for not-a-real-cli` with no signal at all; typos were invisible until `install` / `doctor` later.
**Decision:** warn (not error). Enforced via `drivers::driver_for(name)` with the list of known drivers mentioned in the warning text for discoverability.
**Severity:** minor (UX; no functional bug).

## `add` — unknown CLI in `--cli` silently accepted (Task 10 review)

**Scenario:** `agix add local ./pkg --cli not-a-real-cli` on an Agentfile where `[agix].cli` does not mention `not-a-real-cli`.
**Command:** `agix add <type> <value> --cli <typo>`
**Expected:** exit non-zero; error points at the typo and lists known drivers so the user can correct it.

**Decision:** error (stricter than `check`, which warns). `add` is a write operation and a typo would otherwise persist in the user's TOML.

## `remove` — removing a package that's not in the lock (Task 11 review)

**Scenario:** `agix remove <pkg>` where the package is not tracked by `Agentfile.lock`. Two sub-cases:
  1. Lock file is missing entirely (never installed).
  2. Lock file exists but has no entry for `<pkg>`.
**Command:** `agix remove <pkg>`
**Expected:** surface the fact so the user isn't silently mutating the Agentfile on a typo, but don't pretend work was done that wasn't.
**Actual (before fix):** both cases fell through to `Installer::uninstall → PackageNotFound` via `?`, producing `Error: Package not found: <pkg>`. The manifest edit had already landed on disk (the remove command writes `agentfile_path` before calling the installer), so the user saw a non-zero exit *plus* a mutated Agentfile — inconsistent signal.
**Decision:**
  * Case 1 (no lock file) → warn + success. The manifest edit captures the user's intent; nothing was ever installed so there's nothing to uninstall. Message: `no lock file at <path> — manifest updated, nothing to uninstall`.
  * Case 2 (lock present, no entry) → hard error with actionable message. Distinguishes from Case 1 so a typo isn't silently absorbed. Message: `package '<pkg>' is not in the lock file — nothing to remove`.
**Severity:** minor (UX + CI exit-code consistency).
**Fix commit:** see `review(remove):` commit on this branch.

## `install` — invalid source error didn't name the failing dep (Task 12 review)

## `update` — update-specific silently re-resolved every dep (Task 13 review)

**Scenario:** `agix update <pkg>` on a manifest with multiple deps.
**Command:** `agix update dep-a`
**Expected:** only `dep-a` is re-fetched and re-installed; other deps stay at their locked versions.
**Actual (before fix):** `src/commands/update.rs` called `Installer::uninstall(name)` then `Installer::install_manifest(&manifest, ...)` — the full manifest went back through the resolver, so every dep was re-fetched and its `content_hash` / `sha` could change silently. For local deps the effect was cosmetic (identical bytes re-hashed); for git/marketplace deps it was a hidden upgrade the user didn't ask for.
**Resolution:** `update.rs` now clones the manifest and filters it to just the target dep via `filter_manifest_to` before calling `install_manifest`. Other lock entries are untouched. Regression test: `step3_update_specific_package_refreshes_only_that_dep` in `tests/update_test.rs` asserts `dep-b`'s `content_hash` is unchanged when only `dep-a`'s source mutates.
**Severity:** minor (silent behaviour drift; not a crash).
**Fix commit:** `f9aa064`

## `update` — behaviour when lock file is missing (Task 13 review)

**Scenario:** `agix update` / `agix update <pkg>` against a manifest that has never been installed (no `Agentfile.lock` on disk).
**Command:** `agix update` or `agix update <pkg>`
**Expected:** either treat as install or error with a clear message. Plan (Step 8) explicitly asked us to decide.
**Actual:** update-all silently promotes to a fresh install (iterates the empty lock, does nothing, then calls `install_manifest`); update-specific explodes with `PackageNotFound` via the shared `Installer::uninstall` path.
**Decision:** keep the asymmetry, document it inline, and make the update-specific error message actionable (see finding above). Rationale:
  - Update-all with nothing locked is unambiguously "resolve and install the manifest" — degrading to install is the least-surprising, no-setup-required behaviour.
  - Update-specific with nothing locked is user error: you can't refresh what was never installed. Better to fail loudly and point at the real action (`agix install`).

Regression tests: `step8a_update_all_without_lock_behaves_as_install` and `step8b_update_specific_without_lock_errors_clearly` pin both halves of the decision.
**Severity:** n/a (design decision, not a bug).
**Fix commit:** documented only; inline comment in `src/commands/update.rs` and reference here. Code tidy in `f9aa064`.

## `remove` — `--cli` filter accepts unknown CLI names (Task 11 review)

**Scenario:** `agix remove <pkg> --cli legacy-cli` where `legacy-cli` has no registered driver.
**Command:** `agix remove <pkg> --cli <unknown>`
**Expected:** decide whether to mirror `add`'s strictness (error listing known drivers) or stay lenient for cleanup workflows.
**Actual:** command succeeds; if the Agentfile has a `[<unknown>.dependencies]` section, the entry is removed; otherwise no-op. No warning, no error.
**Decision:** stay **lenient** (intentional asymmetry with `add`). Rationale: `remove` is a cleanup command — users may be deleting legacy `[<old-cli>.dependencies]` sections whose driver was removed from agix. Rejecting unknown CLIs would block that workflow. The design note is captured inline in `src/commands/remove.rs` so future reviewers don't flag it as a bug. If the target section doesn't exist, the removal is simply a no-op for that filter entry. Regression test: `regression_remove_unknown_cli_filter_is_lenient` asserts the command succeeds and the shared dep remains untouched.
**Severity:** n/a (explicit design decision, not a bug).
**Fix commit:** documented-only; see `review(remove):` commit on this branch for the code comment.
